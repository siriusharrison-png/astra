# Application

## Purpose

Application is the step that turns analysis output into decisions or changes for the target project.

Reference analysis is not the end of the work.
It is an input.

## Core Rule

Insights extracted through Astra should be applied to the current project, current task, or current spec.

Do not apply them back into Astra unless the insight is about Astra's own operating rules.

## When to Use

Use application when:

- a reference has already been analyzed
- reusable patterns have been identified
- the current task needs a concrete response
- you need to convert analysis into design, spec, copy, or workflow changes

## Application Types

### 1. Direct Application

Use when the pattern is highly transferable and low risk.

Example:

- a clear content hierarchy
- a useful interaction pattern
- a proven workflow structure

### 2. Adapted Application

Use when the pattern is useful but must be modified for the target context.

Example:

- a visual rhythm that needs a different tone
- a workflow that needs fewer steps
- a thesis-driven page structure that needs different language

### 3. Rejected Application

Use when the reference element does not fit the target project.

Example:

- the tone is wrong
- the interaction is too playful
- the structure is too brand-specific
- the pattern would create confusion or friction

## Application Workflow

1. identify the reusable insight
2. identify the target project need
3. compare fit and risk
4. choose direct, adapted, or rejected application
5. write the result into the target project output

## Output Shape

An application note should usually include:

- source reference
- target project
- insight
- application type
- expected impact
- caution or constraint

## Rule

Application should produce action, not just interpretation.
