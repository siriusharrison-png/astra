# Application

## Purpose

Application is the step that turns analysis output into concrete decisions or changes for the target project.

Reference analysis is not the end of the work. It is an input.

## Relationship to Reference Analysis

If you have followed `protocols/reference-analysis-protocol.md`, you have:
1. Analyzed a reference through 5 phases (Positioning → Structure → Decisions → Language → Transferability)
2. Assessed what is universally useful vs. context-dependent
3. Identified potential constraints

Application takes those findings and decides how to use them in your project.

## Core Rule

Insights extracted through Astra should be applied to the current project, current task, or current spec.

Do not apply them back into Astra unless the insight is about Astra's own operating rules.

## When to Use

Use application when:

- a reference has already been analyzed (or you have clear insights about it)
- you have assessed what is transferable and what is not
- the current task needs a concrete response
- you need to convert insights into design, spec, copy, or workflow changes

## Application Types

Use these three categories to decide how to incorporate the reference insight into your work.

### 1. Direct Application

Use when the pattern is highly transferable and low risk. You can implement it with minimal adaptation.

Example:

- a clear content hierarchy that solves the same problem
- a useful interaction pattern that applies directly to your context
- a proven workflow structure that maps onto your current needs

### 2. Adapted Application

Use when the insight is valuable but must be modified for your context. The principle transfers; the form does not.

Example:

- a visual rhythm that works but needs a different tone
- a workflow that is useful but needs fewer steps
- a structure that fits your goal but requires different language

### 3. Rejected Application

Use when the reference element does not fit your project, even with adaptation. The constraint is too fundamental.

Example:

- the tone is incompatible with your brand
- the interaction assumes a different user context
- the structure is too specific to their situation
- the pattern would create confusion or friction in your use case

## Application Workflow

1. identify the reusable insight (from reference analysis)
2. identify the target project need (what problem are you solving?)
3. compare fit and risk (using the 3 application types above)
4. choose direct, adapted, or rejected application
5. write the result into the target project output

## Output Shape

An application note should usually include:

- source reference (what you analyzed)
- target project (what you're working on)
- insight (what you extracted)
- application type (direct / adapted / rejected)
- expected impact (what changes as a result)
- caution or constraint (what to watch out for)

## Rule

Application should produce action, not just interpretation.

If you have analyzed something but cannot decide how to apply it, the analysis is incomplete—go back and clarify what you learned before attempting application.
