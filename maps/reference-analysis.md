# Reference Analysis

## Purpose

Reference analysis is the process of breaking down a live page, product, or reference artifact into reusable insights that can be applied to a current task.

It is not imitation.
It is selective extraction.

The goal is to identify what is worth borrowing, what is only context-specific, and what should be ignored.
The output should then be applied to the target project or task, not to Astra itself unless the insight is about Astra's own operating rules.

Reference analysis does not end with interpretation.
It should produce a decision for the target project.

## When to Use

Use reference analysis when:

- you have a live site or product you want to study
- you want to extract style, structure, interaction, or messaging patterns
- you need input for a new design, spec, or workflow
- you want to compare a reference against the current requirement

## What to Analyze

### 1. Positioning

- what the page is trying to be
- who it speaks to
- what promise it makes
- what problem it claims to solve

### 2. Structure

- section order
- hierarchy
- repeated patterns
- information grouping
- content density

### 3. Interaction

- navigation
- motion
- state changes
- CTAs
- forms
- playful or experimental elements

### 4. Visual Language

- typography
- spacing
- color strategy
- imagery
- iconography
- texture or atmosphere

### 5. Messaging

- tone
- key claims
- repeated phrases
- value framing
- proof points

### 6. Transferability

For each observed element, decide:

- can it be reused directly?
- can it be adapted?
- should it be treated as reference only?
- should it be ignored?

## Output Shape

A reference analysis should usually include:

- source
- analysis goal
- positioning summary
- structure summary
- interaction summary
- visual language summary
- messaging summary
- transferable patterns
- non-transferable specifics
- recommendations for the current task
- target-project application notes

## Decision Rule

Do not copy everything.

Keep only what:

- fits the current problem
- improves the current direction
- can survive the change of context

If the reference is useful but the target project shape is different, adapt the structure rather than copying the surface.

## Layer Guidance

- Stable layer: how reference analysis should be done
- Working layer: the specific analysis of a live reference
- Evolving layer: recurring extraction patterns that prove useful over time

## Example Use

If a reference page has a strong narrative structure, a clear brand stance, and a distinctive interaction motif, the analysis should separate:

- the stance
- the structure
- the motif
- the visual code
- the portable principle

That separation makes it easier to apply the reference without copying it blindly.

## Structural Borrowing

When a reference is used to inform a project, borrow structure rather than decoration.

Useful structural questions:

- What is the first claim?
- What proof appears next?
- How is the capability organized?
- What is repeated for reinforcement?
- What is the smallest clear path to action?

For a target project, this often means keeping the system modular:

- definition
- principles
- workflow
- maps
- templates
- convergence
- changelog

## Target Project Rule

The output of reference analysis should be routed into the target project as one of these actions:

- direct borrow
- adapted borrow
- reject and ignore

The act of analysis itself should not create new stable Astra content unless the insight changes Astra's own operating rules.
